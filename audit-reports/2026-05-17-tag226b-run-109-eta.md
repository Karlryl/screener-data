## 3. Why Yahoo Pull is the Bottleneck

1. **CDN throttle ceiling.** Yahoo's Cloudflare Edge cannot sustain >8 concurrent quoteSummary requests from a single egress IP without 429-storming (Run #107 = 7,210 rate-limit fails at conc=20). Tag 215f cut to conc=8 to stay under the threshold — but that's a 2.5× throughput cut.
2. **Retry budget compounds the worst-case ticker.** Per `pull-yahoo.js:1130` the retry ladder is `[15s, 45s, 90s]` with a 12s per-attempt timeout. A worst-case ticker (3 retries, all timing out) burns 12+15+12+45+12+90+12 = **198 seconds** of one worker slot. At conc=8 with ~2.3% worst-case rate, that's ~4-5% slot-time lost to retries alone.
3. **Price-only fast-path is the only thing keeping it tractable.** Tag 166's 7-day freshness window means ~80% of daily tickers hit the `yf.quote()` fast-path (~0.4s) instead of full `quoteSummary` (~2.5s steady-state, ~10-200s tail). Without the fast-path, conc=8 against 15.7k full pulls would take **~3300s × 15700 / 8 = ~6.4h** — already over the per-step cap. Run #109's predecessors stayed in budget only because most tickers were warm.
4. **The cliff is universe-doubling.** Tag 222a projected 14.6h linear if every ticker needed a full pull. Today the watchlist is 15,734 (not the 19k Tag 222a modeled). The 165-min per-step cap holds **only because the freshness cache hits 80%+ of tickers**; a cache-cold day (e.g. after a forced cache wipe or a multi-day downtime) would SIGKILL the step.

## 5. Recommended Next Architectural Step: Matrix Sharding

**Problem:** Single-job Yahoo Pull is hard-capped at ~120 min by the Cloudflare-edge throttle. The 240-min job cap can absorb today's 15.7k, but at universe-19k the math breaks even with the price-only fast-path (162 min worst-case Yahoo + 80 min downstream = 242 min, over cap).

**Sketch:** GitHub Actions matrix sharding by `ticker.charCodeAt(0) % N`:

```yaml
jobs:
  pull:
    strategy:
      matrix:
        shard: [0, 1, 2, 3]   # 4 parallel workers, ~3.9k tickers each
    runs-on: ubuntu-latest
    timeout-minutes: 90        # per-shard, was 240 monolithic
    env:
      SHARD_INDEX: ${{ matrix.shard }}
      SHARD_TOTAL: 4
      PULL_CONCURRENCY: '8'    # stays at 8 per-shard — each shard is its own IP
    steps:
      # ... existing steps, but pull-yahoo.js filters watchlist by:
      # if (ticker.charCodeAt(0) % SHARD_TOTAL !== SHARD_INDEX) skip
  merge:
    needs: pull
    runs-on: ubuntu-latest
    steps:
      - name: Merge shard snapshots
        run: node scripts/merge-shards.js
      # ... downstream steps from current single-job
```

**Wins:**
- 4× throughput on Yahoo Pull (each shard has its own runner IP → its own Cloudflare quota).
- 90-min per-shard cap means a single bad shard fails cleanly without taking the others down.
- Downstream steps (which read all snapshots anyway) run in a single merge job, preserving the current single-job invariants (sector-medians, snapshot-picks, walk-forward all need the full universe).

**Costs:**
- Pull workflow becomes 2-job (pull + merge), needs `merge-shards.js` (~30 LOC).
- `_manifest.json` becomes per-shard, then merged at the end.
- More CI minutes total (4× setup + 1× merge), but wall-clock cut from ~120 min to ~30 min.

**Defer until** universe crosses ~17k or until a Yahoo-Pull SIGKILL actually fires. Tag 215f + Tag 219 bought 6-12 months of runway; matrix sharding is the next obvious lever.
