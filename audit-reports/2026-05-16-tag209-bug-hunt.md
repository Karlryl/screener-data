## 1. Sector-medians backward-compat check

PASS — old `median` key is preserved; `_p25_/_p75_/_p90_` enrichment lives
under underscore-prefixed keys that downstream consumers correctly skip.

Evidence:
- `methods/sector-medians-compute.js:120` still writes the bare `result[spId][mid] = median(useArr)`
  before the new lines 126-129 add `_p25_`, `_p50_`, `_p75_`, `_p90_` keys.
- `methods/sector-medians-compute.js:269` (`writeRollingMedians`) explicitly
  skips underscore-prefixed keys: `if (mid.startsWith('_')) continue;` — so
  the new percentile keys do NOT pollute the rolling history.
- `methods/sector-medians-compute.js:323` (CLI summary) filters
  underscore-keys: `Object.keys(m).filter(k => !k.startsWith('_'))`.
- `methods/_helpers.js` `effectiveThreshold()` calls `lookupMedian()` with
  the SPECIFIC `methodId` (e.g. 'roic'), so it reads `result[sp][metricName]`
  — never sees the `_p..` siblings.
- Consumer grep covered: `roic.js`, `fcf-yield.js`, `reinvestment-rate.js`,
  `_helpers.js`. All use `metricName`-keyed reads — none enumerate keys.

No backward-compat regression. Old + new files coexist; only stocks fed to
sector-relative-roic require the new keys (covered in §5).

## 2. Capital-allocation-quality dependency chain

FINDING (MEDIUM) — composite is operating with structurally broken inputs
on every anchor today; no logical bug in the composite itself.

Mechanism:
- `methods/capital-allocation-quality.js:84-87` `require()`s four sub-methods
  at module load — no `allResults` plumbing, no runner ordering dependency,
  no circular-import risk (verified: nobody `require()`s cap-alloc back).
- The composite handles incomputable sub-methods correctly: `_safeEvaluate()`
  catches throws, `computableDimensions` divisor scales the score, and zero
  computable dimensions returns the composite as computable:false.
- BUT live evaluation on 5 anchors reveals the composite is degraded because
  underlying methods don't have their inputs:

  ```
  NVDA: bb=null  debt=25 capex=25 sbc=25  dims=3 → score 100 PASS
  MSFT: bb=null  debt=25 capex=0  sbc=25  dims=3 → score  67 FAIL
  COST: bb=null  debt=25 capex=25 sbc=25  dims=3 → score 100 PASS
  CRDO: bb=null  debt=null capex=25 sbc=0  dims=2 → score  50 FAIL
  PLTR: bb=null  debt=null capex=25 sbc=0  dims=2 → score  50 FAIL
  ```

  `bb=null` on EVERY anchor: buyback-yield needs `annual.annualShares` or
  `timeseries.sharesQ`. Neither is in the current snapshot shape (see
  `methods/buyback-yield.js:58-80`). The buyback-yield method has been live
  since Tag 201 with this same problem; Tag 209c just exposed it.

  `debt=null` on CRDO/PLTR: net-debt-ebitda incomputable for early-stage
  high-growth names without enough EBITDA history.

  CRDO/PLTR run on only 2/4 dimensions (capex + sbc). Scaling them ×2
  yields 50 — which fails the 75 threshold even though they did not actually
  fail any dimension. The 2/4 baseline is too thin to be meaningful — when
  fewer than ~3 dimensions are observable the composite should arguably go
  computable:false rather than emit a confident "fail at 50".

Severity: MEDIUM — DIAGNOSTIC method, not in SCORE_WEIGHTS, fixture-hash
safe. The metric is functionally a `capex+sbc` 2-axis check for any
hypergrowth name and silently misleads (CRDO/PLTR look like capital
mis-allocators when in fact buyback-yield and net-debt-ebitda just don't
have data).

Suggested fix (follow-up tag): in `capital-allocation-quality.js:152-167`,
add a min-dimensions guard. When `computableDimensions < 3`, return
`computable:false` with reason `'only N/4 dimensions observable — composite
not meaningful (need >=3)'`. Surface the contributing values in
`components` for the modal as usual.

Sub-finding (LOW): the comment block at lines 95-98 documents
`BUYBACK_MIN_PCT = 1.0   // buyback-yield returns % units`. Verified —
buyback-yield returns `yieldPct` (line 108 in that file) so 1.0 == 1%
threshold is consistent.

  1. `annualBalance[].accountsReceivable` (Yahoo `netReceivables` in QS)
  2. `annualBalance[].propertyPlantEquipment` (Yahoo `propertyPlantEquipment`)

  4. `annualBalance[].currentLiabilities` (Yahoo `totalCurrentLiabilities`)

     `totalDebt` aggregate)
  6. `annual.annualSGA` (Yahoo `sellingGeneralAdministrative` in IS)
  7. `annual.annualDepreciation` (Yahoo `depreciation` in CF)

Recommendation for the actual promotion tag: extend `mapFTSToBalance()` +
`mapFTSToAnnual()` in pull-yahoo.js to surface these 7 fields, then
backfill anchors, then verify Beneish on NVDA/MSFT/COST/CRDO/PLTR produces
M < -2.22 with margin ≥ 0.5 before flipping the type to DATAGUARD.

Severity: NONE (working as designed — method correctly stays universally
incomputable until the data is there).

## 5. Sector-relative-roic dependency (sector-medians percentile keys)

FINDING (HIGH) — sector-relative-roic returns `computable:false` on every
stock until next sector-medians-compute run.

Mechanism:
- `methods/sector-medians-auto.json` was last regenerated 2026-05-15 at
  13:04 UTC, BEFORE Tag 209b shipped (94211cec4 on 2026-05-16 22:26).
- The file contains 30 sector buckets — ZERO have any `_p25_`/`_p75_`/`_p90_`
  keys (verified by enumerating all bucket keys).
- `methods/sector-relative-roic.js:144` requires both `p50Lookup.value` AND
  `p75Lookup.value` to be non-null. The p50 fallback to `lookupMedian()`
  works (line 138-142), but p75 has no fallback path — file has no `_p75_`
  key, so `lookupPercentile()` returns `not-found` for it.

Live evaluation on the 5 anchors:
```

                        (p50=0.058 p75=null, n>=5 required) — re-run
                        sector-medians-compute to populate

                        (p50=0.076 p75=null, n>=5 required)
COST computable:false  reason: sector percentiles unavailable for CONSUMER_STAPLES
CRDO computable:false  reason: sector percentiles unavailable for HARDWARE
PLTR computable:false  reason: sector percentiles unavailable for SAAS
```

Severity: HIGH — the entire method is dead until the next daily-pull
regenerates sector-medians-auto.json. Header comment at
`methods/sector-relative-roic.js:43-45` does flag this as a documented
failure mode ("pre-Tag-209b generated file, before first re-pull"), but
in practice this means the method ships in a dormant state and the next
daily-pull (cron: `0 6 * * *`) is the activation event.

Note on staleness: rolling-medians file is dated 2026-05-15, 1 day old —
under the 7-day flag threshold. NOT a separate finding.

Suggested fix (follow-up tag): in `methods/sector-relative-roic.js`

`'p75 unavailable — used p50-only ranking'`. This way the method is
useful on day 1 instead of waiting for the cron. (Defer to the daily-pull
