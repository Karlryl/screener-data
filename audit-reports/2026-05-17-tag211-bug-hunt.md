## Findings

### 1. HIGH — analyst-revision-breadth silently treats `null` revisions as zero

**File:** `methods/analyst-revision-breadth.js` lines 93–98 (also 109–127).

**Description.** `Number(null) === 0` is finite. The extractor uses
`Number(row.upLast30Days)` and then `Number.isFinite(u4)` to decide whether
4w data exists. When Yahoo's `earningsTrend` row carries `upLast30Days: null`
but `upLast90Days: 5`, the method treats 4w as `0 − 0 = 0` instead of
`null`, evaluates `net4w = 0`, reports `computable:true`, and produces a
fail (0 < threshold 3) that masks the true unknown.

Reproduced live:

```
input:  { upLast30Days: null, downLast30Days: null,
          upLast90Days: 5,    downLast90Days: 1 }
output: net4w=+0, net12w=+4, pass=false, computable=true   ← WRONG
expect: pass=false (net4w unknown — should be computable=false)
```

**Mechanism.** Lines 93–96 use `Number(row.X)` which coerces `null → 0`,
then line 98 `Number.isFinite(u4) && Number.isFinite(d4)` returns `true`,
so the function reports its first `external.estimateRevisions[pk]` row as
the source even when both 30-day fields are null. The same mistake recurs
on the metrics-shape branch (lines 120–124) where `_unwrap` correctly
returns `null` for `null` but `Number.isFinite(null)` is `false` so that
branch is safe; only Shape 1 is broken.

**Why it matters today.** Latent: no current snapshot carries
`external.estimateRevisions`, so the method universally returns
`computable:false` and the bug never fires in production. It will fire the
moment Tag 21x lands the estimate-revision puller — which is exactly when
the noise will be loudest.

**Suggested fix.** Treat `null`/`undefined` distinctly from `0`:

```js
function _num(v) {
  if (v == null) return null;
  const n = Number(v);
  return Number.isFinite(n) ? n : null;
}
// ...
const u4  = _num(row.upLast30Days);
const d4  = _num(row.downLast30Days);
const u12 = _num(row.upLast90Days);
const d12 = _num(row.downLast90Days);
if (u4 != null && d4 != null) { ... return ... }
```

Then the gate becomes "both 4w counts present, even if both happen to be
the integer 0" — which is the correct semantics.

### 2. MEDIUM — sector-relative-roic degraded path inflates `rank=100` ceiling for any stock at 1.5× sector median

**File:** `methods/sector-relative-roic.js` lines 144–166 (the
documented "degraded p75-missing" fallback) interacting with
`_rankRoic()` lines 94–105.

**Description.** When the auto-medians file lacks `_p75_roic`, the code
substitutes `p75 = p50` (line 163) and documents that "the top-quartile
signal is weaker — synthetic p75 = p50 (so rank>=75 fires when roic>=p50).
Conservative: it never inflates the rank above 75 except at the 100
ceiling." The "100 ceiling" caveat is doing a lot of work — in practice
ANY stock above `1.5 × p50` jumps straight to rank=100, completely
skipping the 75/90 buckets. On the live snapshot:

```
MSFT: ROIC 17.3% vs SAAS p50=7.6% p75=7.6% (degraded) → rank=100
PLTR: ROIC 21.7% vs SAAS p50=7.6% p75=7.6% (degraded) → rank=100
NVDA: ROIC 61.2% vs HARDWARE p50=5.8% p75=5.8% (degraded) → rank=100
```

Three anchors all max out the 0–100 ranking scale on degraded data —
which is the very scale the method exists to differentiate.

**Mechanism.** `_rankRoic(roic, p50, p50)` evaluates the conditions
top-down: `roic >= p75 * 1.5  ⇔  roic >= p50 * 1.5  →  return 100`.
For a SAAS p50 ≈ 7.6 %, that threshold is 11.4 % — virtually every
healthy software anchor clears it.

**Suggested fix.** Either:

(a) Skip the `* 1.5` ceiling in degraded mode — cap at 90:

```js
if (degraded) {
  // Without a true p75, we cannot distinguish "top quartile" from
  // "ceiling decile". Cap the ceiling at 90 so degraded mode emits at
  // most "good — wait for medians refresh".
  const r = _rankRoic(roicValue, p50, p75);
  return r === 100 ? 90 : r;
}
```

(b) Or refuse the degraded fallback for ranks above 75: `computable:false`
when `roic > p50 * 1.5` and `degraded === true`, so the consumer can
choose to ignore until the next sector-medians-compute run.

Either is fine; option (a) is one line and keeps the method useful.

Also cosmetic: `sectorN: p75Lookup.n` ends up `null` in degraded mode,
which the reason string prints as `(n=?)`. Consider sourcing it from
`p50Lookup.n` instead so the consumer still sees the sample-size signal.

### 3. MEDIUM — pull-insider-form4 transient error pollutes per-ticker freshness gate (no auto-retry for 24 h)

**File:** `scripts/pull-insider-form4.js` lines 409–417 (catch block in
the per-ticker loop) interacting with the freshness gate at lines
386–392.

**Description.** When `pullTickerForm4()` throws (timeout, SEC 5xx,
parse error on the submissions JSON, etc.), the catch block writes
`fetchedAt: new Date().toISOString()` onto the per-ticker cache entry
*alongside* a `lastError` field. On the next run, the freshness gate
sees `Date.now() − prev.fetchedAt < 24 h` and unconditionally skips
the ticker — even though we have no transactions for it. The method's
own design promise ("Idempotent / resumable: a ticker's cache entry is
< 24 h old → skip its fetches entirely") is now extending to entries
that *failed*. A transient hiccup silently disables the ticker for a day.

**Mechanism.** Line 411–416:

```js
byTicker[ticker] = Object.assign({}, prev || {}, {
  ticker,
  cik: cikInfo.cik,
  fetchedAt: new Date().toISOString(),     // ← clobbers the TTL
  lastError: e.message
});
```

Line 387:

```js
if (prev && prev.fetchedAt &&
    (Date.now() - new Date(prev.fetchedAt).getTime()) < CACHE_TTL_MS) {
  skippedFresh++; ...
}
```

There is no check on `prev.lastError` or on `transactions.length > 0`
before honouring the TTL.

**Suggested fix.** Two-line guard at the freshness gate:

```js
if (prev && prev.fetchedAt && !prev.lastError &&
    Array.isArray(prev.transactions) &&
    (Date.now() - new Date(prev.fetchedAt).getTime()) < CACHE_TTL_MS) {
  skippedFresh++;
  totalTxns += prev.transactions.length;
  continue;
}
```

Or: in the catch block, write `failedAt` instead of `fetchedAt` so the
freshness gate naturally lets the next run retry.

### 4. MEDIUM — pull-insider-form4 User-Agent contact is a placeholder email

**File:** `scripts/pull-insider-form4.js` line 77.

```js
const USER_AGENT = 'Karl screener-data karl@example.com';
```

**Description.** SEC EDGAR's
[Accessing-EDGAR-Data policy](https://www.sec.gov/os/accessing-edgar-data)
requires the User-Agent to identify the requester with an email address
SEC staff can use to reach them. `karl@example.com` is the IANA-reserved
example-only domain (RFC 2606) and is unreachable. The comment on lines
74–76 acknowledges this: "If SEC ever complains they'll email this
address before blocking." They cannot — the address doesn't exist, so
the first warning step is skipped and the IP block lands immediately.

**Mechanism.** Per SEC policy, non-compliant UA strings (clearly fake
contacts) are auto-flagged and the source IP is throttled or 403'd
without notice. Anchor risk: the daily-pull workflow's IP is shared
across pull-yahoo, pull-sec-xbrl, pull-insider-form4 — a block here
kills all SEC traffic for the whole runner.

**Suggested fix.** Replace with the user's real contact (from the
memory file, `Karl_viehrig@web.de`) or with a project-owned alias:

```js
const USER_AGENT = 'screener-data/1.0 (Karl_viehrig@web.de)';
```

The comment at lines 74–76 should be updated accordingly. Pattern-based;
no per-ticker special-casing.

### 5. LOW — pull-insider-form4 attributes every transaction to the *first* reporting owner

**File:** `scripts/pull-insider-form4.js` lines 220–229, 244–252.

**Description.** `parseForm4Xml` extracts the first `<reportingOwner>`
block and uses it for `personName` / `relationship` on **every** non-
derivative transaction in the filing. Form 4 filings can include
multiple reporting owners (joint trusts, family LLCs, co-trustees);
when they do, all transactions are mis-attributed to the first owner.

**Mechanism.** `_extractFirst(xml, 'reportingOwner')` returns one block;
the per-transaction loop never re-resolves owner from the
`nonDerivativeTransaction` context. There is no SEC schema requirement
that transactions reference an owner, but joint-owner filings are real
and rare — typically <1 % of Form 4s, almost always cleanly-attributed
to a single primary owner anyway.

**Suggested fix.** Defer; tag as future-work. For the consuming methods
(`insider-buy-cluster`, `insider-net-buying`) the aggregate buy/sell
counts and net-shares are unaffected by who is attributed — only the
per-transaction name field is wrong. If the cache becomes the source-
of-truth for a name-level diagnostic, revisit and parse all reporting
owners into a list.
