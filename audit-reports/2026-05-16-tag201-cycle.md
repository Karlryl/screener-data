## Agent 5 — Pre-Commerciality MegaCap Guard (Tag 201b)

Agent 5 found the **smoking gun** of Tag 200's defense-in-depth: when
`annualOpInc[]` is EMPTY (not just zero), `loss-magnitude-guard`,
`metric-divergence-guard`, and `ni-volatility-guard` ALL exit
`computable:false` rather than `pass:false`. The downstream
`!!(method && computable && pass===false)` then evaluates to `false` —
silently bypassing the gate.

Quarantine candidates that slip every existing gate via this bypass:

| Ticker | mcap | rev[0] | annualOpInc | Caught by which existing gate? |
|---|---:|---:|---|---|
| QS | $5.18B | $0 | empty | NONE |
| UPST | $2.66B | $1.04B | empty | NONE |
| LMND | $4.24B | $738M | empty | NONE |
| SOFI | $20.4B | $3.59B | empty | NONE |
| NU | $64.5B | growing | empty | NONE |

**Decision:** add a NARROW guard — `mcap > $1B AND rev[0] < $100M (or null)`.
This catches QS only (and JOBY/ACHR-class pre-revenue $1B+ mega-caps).

UPST/LMND/SOFI/NU are profitable fintech where Yahoo's income-statement
schema differs (financials use different OpInc field names). They should
NOT be hard-gated — that would be a false positive. The fix is a
**pull-yahoo extension** to extract `incomeStatementHistory.totalOperatingIncome`
or sector-aware field-mapping. Deferred to Tag 202+.

**Anchor safety verified by construction:**
- HG/PB anchors all have rev > 100× the $100M floor.
- QC anchors all > 1000× floor.
- Pre-rev micro-caps with mcap < $1B are protected by the precondition.

## Tag 201 Methodology Innovation

The 5-agent parallel pattern is **noticeably more efficient** than
sequential single-agent loops:

- ~6 min wall-clock vs ~30 min for 5 sequential cycles
- Each agent operates with full context window for its scope (no
  cross-domain context dilution)
- Findings can contradict each other safely — orchestrator triages
- Stale-file false-alarm (Agent 1's "null gate flags") was caught
  by orchestrator cross-check, not by the agent itself

**Adopt as default for Tag 202+ cycles.**
