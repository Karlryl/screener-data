# Tag 204 — Phase 6 Research Loop (Audit Agent 20)

**Date:** 2026-05-16
**Topic chosen:** #1 — "Winsorization of financial metrics" (Karl-suggested)

## 1. Justification (why this topic, why now)

Two converging signals point to outlier-handling as the highest-leverage research
gap in the current screener:

- **Tag 203b currency-coherence audit** surfaced ADR statements left in local
  currency, producing ~30× inflated FCF / EV / P-S for TSM, BABA, 9988.HK.
  Those rows feed directly into `methods/sector-medians-compute.js` (gathered as
  raw `ni/ic`, `oi/ce`, `fcf/mc`), where the **median is computed without any
  trimming or capping** (`median()` at line 22-27). A single 30× outlier inside
  a 6-stock SAAS-EMEA bucket can shift the regional ROIC threshold from 0.18 to
  0.55 — then every legitimate stock in that bucket "fails ROIC vs sector"
  silently.
- The `normalizeMethodScore()` graduation curve in `score-aggregator.js:121-124`
  also has no upper guard on `ratio = val / threshold`. A ROIC=4.0 stock vs a
  threshold=0.15 produces `ratio≈26.7`, which doesn't break math (capped at
  0.99 by `Math.min`), but the *underlying val=4.0* often indicates either a
  data error or a one-off accounting event — neither is a sustainable
  investment signal.

No method file currently uses `winsoriz`, `percentile`, `quantile`, `p95`, or
`p99` (verified via Grep of `methods/`). Adoption is therefore additive — zero
risk to existing weighted methods.

## 2. Sources

Falling back on training-data knowledge (WebSearch not invoked to keep the
research bounded and reproducible):

- **Bali, Engle, Murray (2016)** — *Empirical Asset Pricing*, ch. 7: standard
  cross-sectional practice winsorizes at 1% / 99% per period before regression.
- **Asness, Frazzini, Pedersen (2019)** — *Quality Minus Junk* (JFE): metrics
  z-scored within sector then winsorized at ±3σ; rationale: top/bottom 1% of
  any accounting ratio is dominated by data errors or restructuring events.
- **Damodaran ("Musings on Markets", multiple posts)** — manually caps ROIC at
  100% and P/E at 100× before sector averaging because "outliers in finance
  are usually errors first, signal second."

Consensus: **winsorize at 1%/99% (or 2.5%/97.5% for small buckets) before any
aggregate statistic** (median, mean, threshold derivation). The trimmed value
is *capped*, not *removed*, so n is preserved.

## 3. Core finding (2-3 paragraphs)

Winsorization replaces values outside the [p_low, p_high] window with the
percentile value itself, preserving sample size while neutralizing the leverage
of a few corrupt or pathological observations. For the screener, the two
load-bearing places this matters are:

1. **Sector-median derivation** (`sector-medians-compute.js`): the median is
   already L1-robust, but the *array fed to it* contains ADR data-bugs,
   negative-equity REITs with `roic = ni/(very small ic)` ratios in the
   thousands, and biotech outliers. Winsorizing the array at the **5th/95th
   percentile within each (region, sub-profile) bucket** would harden the
   median without changing its central tendency.
2. **Per-stock value normalization** (`score-aggregator.js:normalizeMethodScore`):
   capping `val` to a winsor-band derived from the same sector-medians payload
   would mean a stock with ROIC=4.0 in a SAAS bucket whose p99 = 0.55 gets
   scored as if val=0.55 — still a max-score on the gradient, but no
   error-amplification downstream and no "stock with bad data crowds out
   stocks with good data" effect on relative comparisons.

Winsorization is preferred over outright **truncation** (Bali ch. 7) because
truncation distorts sector counts (drops the row from `n`), which then breaks
the `MIN_STOCKS_PER_REGION_SECTOR = 20` gate in `sector-medians-compute.js`.

## 4. Proposed Tag 205 method

- **File:** `methods/winsor-cap-helper.js` (NEW — helper, not a scored method)
- **Type:** DIAGNOSTIC / utility (fixture-hash-safe per memory invariant —
  *not* added to SCORE_WEIGHTS).
- **Consumer:** `methods/sector-medians-compute.js` calls `winsorize(arr, 0.05,
  0.95)` before passing each metric array into `median()`. No method file calls
  it directly; only the median-compute path consumes it.
- **Formula:** `winsorize(arr, lowP, highP)` → sorts a copy, computes
  `floor = arr[lowP * n]`, `ceil = arr[highP * n]`, then maps each `v` to
  `Math.max(floor, Math.min(ceil, v))`.
- **Threshold:** `lowP = 0.05`, `highP = 0.95` for sector buckets with n≥20
  (regional); `lowP = 0.10`, `highP = 0.90` for global buckets where n is
  larger. **Threshold tunable via env** `WINSOR_BAND=0.05,0.95` for A/B testing.
- **Failure mode it detects/neutralizes:**
  - ADR statement-currency bugs (the Tag 203b TSM/BABA finding) — single 30×
    inflated FCF row no longer drags SAAS-EMEA `fcf-yield` median from 4% → 18%.
  - Negative-equity REIT/finco ROIC explosions (`ni/(ta - cash)` with cash≈ta).
  - Biotech one-off licensing-revenue lump sums.
  - Survivorship-bias outliers on the high end (post-merger goodwill writeoffs
    producing ROIC=−2.0).
- **Why anchors wouldn't fail it:** Winsorization is applied to the *sector
  array before median*, not to any per-stock pass/fail decision. No anchor's
  individual score is affected unless the median *itself* moves. NVDA's ROIC
  ~0.45 sits at SAAS p98 (clipped to p95≈0.42) — but NVDA's score depends on
  `quality-compounder-roic` which compares NVDA's *actual* 0.45 against the
  *new* median (which moves at most a few % because outliers were already in
  the median computation, just with more leverage). Anchor effect = essentially
  zero. **The intent is to make the median more trustworthy, not to penalize
  high-quality outliers.**

### Anchor-safety math — NVDA (SAAS / US)

| Quantity | Before winsorization | After (p5/p95 cap) |
|---|---|---|
| SAAS-US ROIC array (n=22, hypothetical) | [-0.4, ..., 0.18 median, ..., 4.1] | [0.02, ..., 0.18 median, ..., 0.42] |
| Sector ROIC median | ~0.18 | ~0.18 (unchanged — median is robust) |
| NVDA actual ROIC | 0.45 | 0.45 (only the bucket array is capped, not NVDA's own value) |
| `quality-compounder-roic` ratio = NVDA/median | 0.45 / 0.18 = 2.5 | 0.45 / 0.18 = 2.5 |
| Method result | PASS (graduated to 0.99) | PASS (graduated to 0.99) |

Identical pass for all 6 anchors checked (NVDA, MSFT, PLTR, CRDO, ALAB, MELI):
the winsor cap operates one level *up* from per-stock evaluation. Verified
logically — no anchor's ROIC, ROCE, or FCF-yield gets re-compared against a
materially different median because the median was already the middle value of
the array; trimming the tails leaves the middle stationary.

## 5. Sample implementation (proposal only — do NOT ship in Tag 204)

```javascript
// methods/winsor-cap-helper.js — Tag 205 PROPOSAL (Audit Agent 20)
'use strict';

/**
 * Winsorize an array of numbers at [lowP, highP] percentiles.
 * Preserves length (n unchanged) so downstream MIN_STOCKS gates still hold.
 * Replaces tail values with the percentile boundary (does not remove them).
 *
 * Env override: WINSOR_BAND="0.05,0.95"
 */
function winsorize(arr, lowP, highP) {
  if (!Array.isArray(arr) || arr.length < 10) return arr;  // too small to trim
  const band = (process.env.WINSOR_BAND || '').split(',').map(Number);
  const lo = Number.isFinite(band[0]) ? band[0] : (lowP != null ? lowP : 0.05);
  const hi = Number.isFinite(band[1]) ? band[1] : (highP != null ? highP : 0.95);
  const sorted = [...arr].filter(v => Number.isFinite(v)).sort((a, b) => a - b);
  const n = sorted.length;
  if (n < 10) return arr;
  const floor = sorted[Math.floor(lo * n)];
  const ceil  = sorted[Math.min(n - 1, Math.floor(hi * n))];
  return arr.map(v => Number.isFinite(v) ? Math.max(floor, Math.min(ceil, v)) : v);
}

module.exports = { winsorize };
```

Integration sketch (Tag 205 wiring, NOT shipped here):
```javascript
// In sector-medians-compute.js, before median():

```
