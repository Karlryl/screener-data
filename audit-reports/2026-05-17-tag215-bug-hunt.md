## 3. Findings

### HIGH-1 — `analyst-revision-breadth.js`: period priority systematically picks lowest-sample window

**File:** `methods/analyst-revision-breadth.js` line 104 (`periodKeys`).
**Description:** The method iterates `['0q','+1q','0y','+1y']` and uses the
**first** period whose 30-day window is non-null. In every anchor snapshot
checked, that's `0q` (current quarter). `0q` covers fewer analysts than
`0y`/`+1y` and is the noisiest window.

Evidence (MSFT, 2026-05-17 snapshot):

| Period | up30d | down30d | net4w |
|---|---|---|---|
| 0q     | 8     | 17      | **-9**  |
| +1q    | 11    | 7       | +4   |
| 0y     | 19    | 0       | +19  |
| +1y    | 18    | 9       | +9   |

The 0q-first rule yields MSFT net4w=-9 (matching the audit-prompt's expected
anchor), but the same rule systematically discards the larger-sample 0y/+1y
data that better captures consensus direction.

**Mechanism:** First-match-wins, with `0q` always populated by Yahoo for any
covered name, means the larger windows are never reached.

**Suggested fix (pattern-based):** Either (a) sum nets across all available
periods, weighted by max(up+down), (b) prefer `0y` (full-year) when present
and fall back to `0q` only for newly-covered names, or (c) emit
per-period breakdown in components and compute net4w as the average of
non-null periods. Document the period-selection trade-off in the method
header — current header lists priority without justification.

### HIGH-2 — `fcf-conversion-stability.js`: 0.85 threshold mis-calibrated against documented anchors

**File:** `methods/fcf-conversion-stability.js` line 71 (`THRESHOLD = 0.85`).
**Description:** Header documentation (lines 60-62) claims:

> MSFT FCF/NI typically 0.95-1.10 → PASS.
> GOOG ~0.85-1.05 → PASS (borderline some years).

Live anchor results (today):

| Ticker | geomean | pass | ratios (latest-first) |
|---|---|---|---|
| MSFT | **0.812** | **FAIL** | 0.70, 0.84, 0.82, 0.90 |
| GOOG | **0.785** | **FAIL** | (4y) |
| NVDA | 0.854 | PASS (barely) | |
| MELI | 4.685 | PASS (huge) | 5.39, 3.69, 4.69, 5.16 |

Both MSFT and GOOG — paradigmatic quality compounders — fail. MELI's
4.7x geomean (driven by 5x FCF/NI ratios) suggests the geomean is
dominated by exceptional years rather than discriminating low-quality
from high-quality firms.

**Mechanism:** Either (a) the 0.85 threshold is too tight for the
current data window (MSFT/GOOG capex up-cycle); or (b) the geometric mean
on uncapped ratios over-rewards lumpy-positive years while under-
penalising chronic under-conversion.

**Suggested fix:** Either lower threshold to 0.75 (matches MSFT's worst
4y window), or cap individual ratios at 2.0 before the geomean to reduce
outlier-amplification; verify against a 20-name basket before flipping.
Update header anchor claims to match observed reality.

### MEDIUM-1 — `pull-yahoo.js` line 1188: Tag 211l adds SGA/Depreciation without bumping FTS_CACHE_VERSION

**File:** `pull-yahoo.js` line 1188 (`FTS_CACHE_VERSION = 2`) +
1218-1219 (`ftsAnnualSGA = cached.payload.ftsAnnualSGA || []`).
**Description:** Tag 211l added `ftsAnnualSGA` and `ftsAnnualDepreciation`
to the cache payload but did not bump `FTS_CACHE_VERSION`. Comment at
line 1215 acknowledges:

> Old caches will return undefined → default to empty array. Stocks get
> these fields after their cache expires (CACHE_TTL_MS) and re-pulls.

`CACHE_TTL_MS = 28 * 86400 * 1000` — up to 28 days of silently-degraded
sga-revenue-trend and beneish-m-score evaluations for any ticker whose
FTS cache is still on the old payload shape.

**Mechanism:** Until each ticker's cache expires, `cached.payload.ftsAnnualSGA`
is undefined → coerced to `[]` → `canonical.annual.annualSGA` never assigned
→ `sga-revenue-trend` returns `computable=false` and `beneish-m-score`
returns `missingFields=[annualSGA, annualDepreciation]`. The screener
silently reports incomputable instead of showing real data for ~28 days.

**Suggested fix:** Bump `FTS_CACHE_VERSION` to 3. Forces immediate refresh
on all tickers; old caches rejected per line 1195. One-shot cost of fresh
pulls; gains correct data immediately.

### MEDIUM-2 — `ohlson-o-score.js` lines 145-146 / 163 / 175: `wcDirect` fallback partially broken

**File:** `methods/ohlson-o-score.js`.
**Description:** Line 145-146 allows the missing-field check to pass when
`wcDirect` is provided (even if `ca_t`/`cl_t` are null). Line 161 then
derives `wc_t = wcDirect`. But line 175 unconditionally uses
`CL_CA = 0.0757 * (cl_t / ca_t)` — `null/null` → NaN, then line 200
`!Number.isFinite(O)` → `_missing("O-Score not finite")`.

Test case verified live:

```js
{ totalAssets: 5e9, totalLiabilities: 2e9,
  currentAssets: null, currentLiabilities: null, workingCapital: 5e8 }
```

Returns `computable: false, reason: 'O-Score not finite (NaN/Inf in sub-term)'`
instead of computing successfully with the supplied wcDirect.

**Mechanism:** Ohlson's CL_CA term cannot be derived from wcDirect alone —
both CA and CL are needed individually. The wcDirect fallback was intended
to allow snapshots that pre-compute WC without exposing CA/CL.

**Severity LOW currently** because no live snapshot exposes `workingCapital`
without also exposing `currentAssets`/`currentLiabilities`. Latent — fires
if a future puller (or external data source) follows the inverse pattern.

**Suggested fix:** Either (a) require ca/cl to be present alongside wcDirect
in the missing-field check (drop the `&&` collapse), or (b) accept wcDirect
only when CA and CL are both finite (drop the special case entirely — derive
WC from CA-CL when both are present).

### MEDIUM-3 — `institutional-ownership-13f.js` line 127: `value: 0` returned with `computable: false`

**File:** `methods/institutional-ownership-13f.js` lines 125-132.
**Description:** When the ticker is absent from the cache, the method
returns `value: 0` alongside `computable: false`. The other incomputable
branches (no cache file, no ticker identifier) correctly omit `value`
(letting `buildResult` clamp to null).

UI consumers reading `value` directly (without checking `computable`)
will see "0 institutions" when the actual state is "no data". The
inconsistency between branches is the issue — either always emit
`value: 0` for missing-data cases or never.

**Mechanism:** Cosmetic / semantic; downstream consumers of `.value`
without `.computable` guard get a misleading numeric.

**Suggested fix:** Remove `value: 0` from the ticker-not-in-cache branch
(line 127). All 4 incomputable returns then consistently emit `value: null`
via `buildResult`'s finite-check.

### LOW-2 — `fcf-conversion-stability.js` `negativeRatioFloorUsed` correct but fragile

**File:** `methods/fcf-conversion-stability.js` line 156.
**Description:** `ratios.some((r, i) => rawRatios[i] !== r)` correctly
detects whether floor was applied by comparing parallel arrays. Works
because rawRatios and ratios are pushed in lockstep within the same loop
(lines 118-121). If a future refactor decouples the two pushes, the
positional alignment breaks silently. Add an inline comment asserting the
invariant.

### LOW-3 — `earnings-power-stability.js` line 127: population stdev (divisor n) not sample stdev (n-1)

**File:** `methods/earnings-power-stability.js` lines 127-128.
**Description:** Uses population stdev: `variance = sum / n`. QMJ/Lepetit
literature typically uses sample stdev (n-1 divisor) for stability metrics.
With n=4, this gives CoV that is `sqrt(4/3) = 1.155×` smaller than the
sample-stdev formulation — i.e. systematically more passes.

Not necessarily wrong (population stdev is defensible for "the only sample
we have"), but inconsistent with academic convention. `revenue-quality-cov`
has the same convention (line 144, divisor n).

**Suggested fix:** Either switch to sample stdev (`/ (n-1)`) to match QMJ
convention and re-anchor the threshold, or document the choice in the
header. Same fix applies to revenue-quality-cov.
